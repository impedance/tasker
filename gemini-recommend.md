# Архитектурный взгляд и рекомендации по MVP Tasker

Проанализировав исходные документы проекта (PRD, RFC и структуру Эпиков), я подготовил ряд рекомендаций по приоритетам реализации и выбору технического стека. Данный файл резюмирует выводы и предложения, которые помогут запустить проект быстрее и качественнее.

## 1. Сильные стороны проекта (что сделано отлично)

*   **Жесткие границы MVP:** отказ от полноформатного бэкенда, мультиплеера, процедурной генерации карт и сложного AI. Это действительно поможет сфокусироваться на проверке ключевой гипотезы и защитит проект от бесконечного затягивания.
*   **Архитектурное разделение логики:** вынесение правил (доменной логики изменения состояний, тумана войны, осад) в чистые функции и формальное отделение хранилища от UI — это залог того, что код не превратится в кашу на тесно связанных React-компонентах. В будущем это значительно упростит добавление облачной синхронизации.
*   **Четкая система эпиков:** разбивка на 13 эпиков очень проработанная, структурная и последовательная. План выглядит как готовый roadmap, который возможно реализовать за 3 недели интенсивной разработки.

## 2. Предлагаемые изменения приоритетов (Scope MVPs)

*   **Онбординг и Демо-проект (в PRD сейчас P1):** Рекомендую настоятельно **поднять приоритет до P0 (must-have)**. UI игры сильно отличается от привычных менеджеров задач (Notion, Todoist). При виде пустой карты новый пользователь может не понять, что делать, и уйти. Нужна заранее сгенерированная "кампания-туториал", где пользователь за 1 минуту "снимет туман" и "отобьет осаду". Это критично для понимания механик игр и метрики *D1 retention*.
*   **EPIC-09 (Adaptation - Rule-based adaptation):** Понизить приоритет до **P1** или даже **P2**. Для самого первого MVP сложный движок рекомендаций на основе паттернов поведения пользователя — избыточен. Для старта достаточно статичной привязки тактик к состояниям или легкой рандомизации базовых подсказок. Понижение приоритета сэкономит 1-2 дня.
*   **EPIC-11 (Scoring-feedback):** Систему очков на первом этапе можно сильно упростить, оставив просто визуальный прогресс-бар для провинции. Сложная балансировка очков и защита от абуза (anti-abuse rules) не так критичны для проверки основной механики. Спорные моменты лучше решить "сурогатом" в виде визуального похлопывания по плечу.
*   **EPIC-12 (Instrumentation):** В офлайн-приложении сбор логов в JSON для аналитики может быть полезен автору, но это стоит делать в самом конце, если останется время, или вообще перенести в версию продукта v0.2.

**Порядок разработки:** Сначала стоит сфокусироваться на EPIC-01 .. EPIC-08, так как без ежедневного цикла (Daily loop) и тактик осад (Siege) продукт не имеет смысла: эта механика — ядро проекта. EPIC-09, 10, 11 и 12 можно смело "резать" или упрощать, если времени не будет хватать к третьей неделе.

## 3. Оценка технического стека

Стек **React + TypeScript + Vite + Zustand + localForage + SVG** подобран идеально для концепции:

1.  **React + TS + Vite:** Классический набор для быстрого прототипирования сложных клиентских интерфейсов. TypeScript защитит доменные правила и контракты состояний (state machine) от глупых ошибок, а Vite даст молниеносную скорость разработки HMR.
2.  **Zustand:** Идеально подходит. Redux был бы слишком громоздким (много boilerplate кода) и медлительным для старта, а Context API недостаточно производительным для часто меняющегося игрового состояния, особенно с привязкой к постоянному сохранению.
3.  **localForage (IndexedDB):** Надежное решение для хранения данных браузера. Обычный `localStorage` мог бы столкнуться с ограничениями размера в 5MB и блокировкой потока UI. `localForage` дает асинхронный API (promises) и позволяет надежно работать с IndexedDB.
4.  **SVG + CSS animations для интерактивной карты:** Очень мудрое решение. SVG позволит использовать стандартные встроенные React-события (onClick / onMouseEnter и др.), легко окрашивать фрагменты карты через CSS классы (статусы провинций) и без особых усилий привязывать HTML/DOM Tooltip-ы поверх интерфейса, что было бы на порядок сложнее реализовать с использованием "тяжеловесных" Canvas или Phaser. Геометрия карты — это по сути просто кликабельные нерегулярные секции.
5.  **Тестирование (Vitest + RTL + Playwright):** Ровно тот оптимальный объем QA, который обеспечит стабильность "чистых функций" домена (Vitest) и гарантирует, что пользователь без краша может пройти свой главный пользовательский путь от создания кампании до закрытия задачи (Playwright).
